This is what an instructor gives you. Read it carefully — your job is to think critically about it before the AI starts working.

Raw Problem Statement

Design a real-time sorting module for an IoT sensor network. The system receives readings from a few hundred temperature sensors. It must sort recent readings by timestamp, handle data spikes during peak hours, and support a polling API that returns the latest sorted window. Optimize for low memory overhead.
What's happening: In a real challenge, this is the first thing you see. The problem sounds reasonable, but is it complete enough for an AI to solve well? That's what you'll figure out next.

Notice what's missing?

Real-world problems are often ambiguous. A key co-reasoning skill is spotting the gaps before the AI starts solving.

The problem again

Design a real-time sorting module for an IoT sensor network. The system receives readings from a few hundred temperature sensors. It must sort recent readings by timestamp, handle data spikes during peak hours, and support a polling API that returns the latest sorted window. Optimize for low memory overhead.
  • Sort key is unclear: "Sort by timestamp" — but what about readings with identical timestamps? There is no tiebreaker strategy.
  • "Few hundred" is vague: Is it 200? 500? 900? The data structure and algorithm choice depends heavily on actual scale.
  • "Data spikes" is undefined: Does this mean 2x the normal rate? 100x? Should the system drop readings, buffer them, or handle all of them?
  • No window size specified: "Latest sorted window" — how large? Last 10 seconds? Last 1000 readings? This changes the memory model entirely.
This is Framing: Before the AI writes a single line of code, you refine the problem by identifying what's missing, ambiguous, or under-specified. Better framing leads to better AI output.

Frame it

Select the statements that correctly identify gaps or add useful constraints. In a real challenge, you pick from multiple-choice options.

✓
The sort key needs a tiebreaker — when timestamps match, use sensor ID as secondary sort.
Correct: Identifies a genuine ambiguity and proposes a concrete resolution.
✓
Define "few hundred" as exactly 500 sensors, each sending one reading per second.
Correct: Pins down a vague quantity with a specific, testable number.
✗
Use Python because it's the most popular language for IoT.
Incorrect: This is an implementation choice, not a problem clarification. Framing is about the problem, not the solution.
✓
Define "data spike" as up to 5x normal throughput and specify that no readings should be dropped.
Correct: Clarifies an ambiguous term with concrete parameters and a clear requirement.
✗
The problem is clear enough — no additional constraints are needed.
Incorrect: The problem has several ambiguities that would lead to an incomplete solution.
Scoring: You earn a higher Framing grade by correctly identifying real gaps and avoiding options that jump ahead to solutions or dismiss the need for clarification.

AI generates a solution

Based on your framing, the AI produces a solution. But AI solutions are often plausible yet flawed — your job is to evaluate it critically.

AI-Generated Solution

Approach: Use a fixed-size circular buffer of 1000 entries, sorted via insertion sort on each new reading.

class SensorSorter:
    def __init__(self, window=1000):
        self.buffer = []
        self.window = window

    def ingest(self, reading):
        # Insert in sorted position by timestamp
        pos = bisect.bisect_left(
            [r.timestamp for r in self.buffer],
            reading.timestamp
        )
        self.buffer.insert(pos, reading)
        # Trim to window size
        if len(self.buffer) > self.window:
            self.buffer.pop(0)

    def get_sorted_window(self):
        return self.buffer[:self.window]

Complexity: O(n) per insertion due to list shift. Memory is bounded at O(window_size).

Look carefully: This solution compiles and sounds reasonable, but it has real problems. Can you spot them? That's the next step.

Judge the solution

Identify what's wrong with the AI's solution. Select the statements that correctly point out real issues.

✓
The list comprehension [r.timestamp for r in self.buffer] rebuilds the entire key list on every insertion — this is O(n) overhead on top of the O(n) insert.
Correct: This is a genuine performance bug. The key extraction should use a separate sorted structure or SortedList.
✗
The circular buffer concept is fundamentally wrong for this use case.
Incorrect: A bounded buffer is a reasonable approach for a sliding window — the implementation details are the problem, not the concept.
✓
There is no tiebreaker when timestamps are identical — sensor ID should be used as a secondary sort key.
Correct: This was identified during framing but the AI ignored it in the implementation.
✓
During data spikes (5x throughput), the O(n) insertion will cause latency buildup with 500 sensors sending simultaneously.
Correct: At 2500 readings/sec during spikes, O(n) per insert on a 1000-element list creates a real bottleneck.
✗
Python is too slow for real-time systems; this should be rewritten in C++.
Incorrect: While language choice matters, judging is about evaluating the solution's logic and correctness, not making implementation decisions.
Scoring: You earn a higher Judging grade by finding real logical and correctness issues. Avoid vague complaints or opinions about technology choices.

Steer the AI

Now guide the AI to fix its mistakes. Select the corrections that are specific and actionable.

✓
Replace the list + bisect approach with a SortedList from the sortedcontainers library, which gives O(log n) insertion and avoids rebuilding the key list.
Correct: Specific fix, names the exact library, and explains why it solves the performance issue.
✗
Make the sorting faster.
Incorrect: Too vague — the AI needs to know what to change and how. "Faster" doesn't guide anything.
✓
Modify the sort key to be a tuple (timestamp, sensor_id) so that readings with identical timestamps are ordered deterministically.
Correct: Precise instruction — specifies the exact data structure change and the reason for it.
✓
Add a spike detection mechanism: when ingest rate exceeds 2500/sec, switch to batch insertion (collect for 200ms, then bulk-sort) to amortize overhead.
Correct: Addresses the spike scenario with specific thresholds and a concrete strategy.
✗
The solution has too many bugs, please start over from scratch.
Incorrect: Throwing away work isn't steering — good corrections build on what exists and fix specific issues.
Scoring: You earn a higher Steering grade by giving precise, constructive corrections. Tell the AI exactly what to change and why — not just "fix it."
